`s@`: Decentralized Social Networking over Static Sites

Modern social networking platforms are architecturally converging on the same solution: large-scale distributed databases with firehoses, relay networks, and centralized identity providers. Even decentralized alternatives such as ActivityPub (Mastodon) and the AT Protocol introduce servers or relay infrastructure that become new points of centralization and failure.  We argue this complexity is unnecessary for the core use case that social networking was intended for -- keeping up with your friends -- and present `s@`, a decentralized social networking protocol built entirely on static file hosting.

In `s@`, each user's social graph, posts, and encryption keys reside in a static directory served over HTTPS. There are no dedicated servers or message-passing relays. Identity is authenticated by TLS: possession of a domain name is sufficient proof of authorship. A browser-side client aggregates feeds by directly fetching encrypted files from each followed user's site, decrypting them locally, and merging by timestamp. Publishing is equally direct: a new post is encrypted and pushed to the user's own static host via a storage API (e.g., GitHub Contents API).

The encryption model is noteworthy. Content is encrypted with a per-user symmetric key, and the content key is distributed to each follower via an individually sealed box. This eliminates the need for a key server: the user's own static site is the key distribution point. Key rotation on unfollow is a straightforward local operation -- regenerate the content key, re-encrypt all posts, and update follower envelopes -- enforcing forward-access revocation without any coordination with a third party.

`s@` deliberately does not scale beyond the social graph of a real human being. A user follows handfuls of friends, and the client fetches that many sites. This is a feature: it re-centers the design on the actual workload rather than on the infrastructure required to support viral content distribution to millions. The result is a system with no operational cost beyond static file hosting, no administrator, no moderation surface, and no algorithmic feed -- yet one that supports the full cycle of posting, following, feed aggregation, threading, and access-controlled encryption.

We present the protocol design, the encryption scheme, and a sample implementation running entirely in the browser. We discuss the systems trade-offs of treating a CDN as a database, the limitations of static-site consistency for social applications, and the surprising degree to which the constraints of static hosting simplify the security model.
