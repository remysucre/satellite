s@: Decentralized Social Networking over Static Sites

Modern social networking platforms are architecturally converging on the same solution: large-scale distributed databases with firehoses, relay networks, and centralized identity providers. Even decentralized alternatives such as ActivityPub (Mastodon) and the AT Protocol introduce servers or relay infrastructure that become new points of centralization and failure.  We argue this complexity is unnecessary for the core use case that social networking was intended for — keeping up with your friends — and present s@, a decentralized social networking protocol built entirely on static file hosting.

In s@, each user's social graph, posts, and encryption keys reside in a static directory served over HTTPS. There are no dedicated servers, no message-passing relays, and no consensus protocols. Identity is authenticated by TLS: possession of a domain name is sufficient proof of authorship. A browser-side client aggregates feeds by directly fetching encrypted files from each followed user's site, decrypting them locally, and merging by timestamp. Publishing is equally direct: a new post is encrypted and pushed to the user's own static host via a storage API (e.g., GitHub Contents API).

The encryption model is noteworthy. Content is encrypted with a per-user symmetric key (XChaCha20-Poly1305), and the content key is distributed to each follower via an individually sealed box (X25519 + libsodium crypto_box_seal). This eliminates the need for a key server: the user's own static site is the key distribution point. Key rotation on unfollow is a straightforward local operation — regenerate the content key, re-encrypt all posts, and update follower envelopes — enforcing forward-access revocation without any coordination with a third party.

The resulting system has an unusual transaction profile. Reads are fully parallel and embarrassingly local to the client. Writes are serialized only by the publishing API of the user's own host, and cross-user write coordination does not exist. There are no distributed transactions and no global ordering requirements. The "database" is an append-only log of encrypted blobs, with a plaintext index file acting as a lightweight manifest. Consistency is eventual in the sense that a new post becomes visible to followers only after they re-fetch the index, but this is not a bug — it matches the delivery semantics users already expect from social feeds.

s@ deliberately does not scale beyond the social graph of a real human being. A user follows handfuls of friends; the client fetches that many sites. This is a feature: it re-centers the design on the actual workload rather than on the infrastructure required to support viral content distribution to millions. The result is a system with no operational cost beyond static file hosting, no administrator, no moderation surface, and no algorithmic feed — yet one that supports the full cycle of posting, following, feed aggregation, threading, and access-controlled encryption.

We present the protocol design, the encryption scheme, and a sample implementation running entirely in the browser. We discuss the systems trade-offs of treating a CDN as a database, the limitations of static-site consistency for social applications, and the surprising degree to which the constraints of static hosting simplify the security model. s@ offers a case study in right-sizing infrastructure to workload: when the workload is friendship, a static file server is enough.
